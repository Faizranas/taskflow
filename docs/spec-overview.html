TaskFlow — Spec-Driven Development Overview

Why every markdown file exists, how they interlink, and what value they deliver to Claude Code

Why So Many Files?

🎯

One File = One Responsibility

Each file has a single, clear job. PRD defines WHAT. TECHNICAL_SPEC defines HOW. CLAUDE.md defines RULES. Specs define CONTRACTS. Tasks define WORK ORDERS. Mixing these would make every file bloated and untrustworthy.

🪙

Token Efficiency for AI

Claude Code has a context window. Sending everything on every task wastes tokens and causes confusion. By splitting files, you send only what's needed per task — e.g. auth task only gets CLAUDE.md + specs/auth.md.

🔒

Single Source of Truth

If a rule lives in one place, it can only be wrong in one place and fixed in one place. Repeating rules across files causes drift — one copy gets updated, others don't, and AI produces inconsistent code.

📋

Auditability

Every implementation decision traces back to a spec file. If something in the code is wrong, you find the spec, fix it, and re-run the task. No guessing what the AI was thinking.

Every File Explained

📁 docs/ — Reference Documents (read by humans + AI at module start)

docs/PRD.mdHuman writes

What it is: Product Requirements Document. Defines features, user stories, business rules, and what is out of scope for v1.

Why it exists: Establishes the "why" and "what" before any technical decisions. Prevents scope creep.

Read by: Humans only. Never sent to Claude Code.

docs/TECHNICAL_SPEC.mdHuman writes

What it is: Full architecture document. Contains ERD (Mongoose schemas), folder structure, API patterns, auth flow, error handling, and environment config.

Why it exists: This IS the architecture document. Claude Code reads it before starting any new module to understand the full system design.

Read by: Claude Code at task 001 and when starting a new module.

📄 CLAUDE.md — AI Conventions (sent with every single task)

CLAUDE.mdHuman + AI

What it is: Slim (~400 token) standing rules file. Covers stack, naming conventions, folder rules, TypeScript rules, architecture patterns, Mongoose rules, testing rules, and hard constraints.

Why it exists: Claude Code starts fresh on every task. Without this file it would make different decisions each time — inconsistent naming, wrong patterns. This file is Claude Code's permanent memory.

Read by: Claude Code on every task without exception.

📋 specs/ — Module Contracts (sent only for relevant tasks)

specs/auth.mdHuman writes

What it is: Complete auth contract. Defines Mongoose schemas, all 4 endpoints with exact request/response shapes, validation rules, JWT utility signatures, middleware spec, and 10 test cases.

Why it exists: Auth is the foundation of everything. If any detail is missing the AI will invent it — and invent it inconsistently.

Read by: Claude Code only during tasks/002-auth.md

specs/projects.mdHuman writes

What it is: Projects module contract. Defines Mongoose schema, 5 endpoints, cascade delete behaviour, Zod validation schemas, and 12 test cases.

Why it exists: Without it Claude Code doesn't know about cascade deletes, partial update rules, or the correct sort order for project lists.

Read by: Claude Code only during tasks/003-projects.md

specs/tasks.mdHuman writes

What it is: Tasks module contract — the most complex spec. Defines Mongoose schema with 5 indexes, 5 endpoints, 7 filter parameters, tag population rules, Zod schemas, and 18 test cases.

Why it exists: Filtering logic is highly specific. Without a spec the AI cannot know that dueBefore maps to $lte or that tags must always be populated — never raw ObjectIds.

Read by: Claude Code only during tasks/005-tasks.md

specs/tags.mdHuman writes

What it is: Tags module contract. Defines schema with unique constraint, 4 endpoints, duplicate name check, $pull behaviour on delete, and 12 test cases.

Why it exists: The $pull vs $unset distinction and the "tags belong to user not project" design decision must be explicit — otherwise AI guesses wrong.

Read by: Claude Code only during tasks/004-tags.md

⚡ tasks/ — AI Work Orders (one per implementation session)

tasks/001-project-setup.mdHuman writes

Scaffolds entire project skeleton: folders, package.json, tsconfig, app.ts, db.ts, env.ts, AppError, asyncHandler, errorHandler, frontend HTML. No business logic.

tasks/002-auth.mdHuman writes

Implements full auth module by referencing specs/auth.md. Points Claude Code at the right spec sections without repeating content.

tasks/003-projects.mdHuman writes

Implements projects module. References specs/projects.md. Defines route mounting instruction for app.ts.

tasks/004-tags.mdHuman writes

Implements tags module. References specs/tags.md. Tags must be complete before tasks since tasks reference tag IDs.

tasks/005-tasks.mdHuman writes

Implements tasks module — most complex task. References specs/tasks.md. Depends on projects + tags being done first.

tasks/006-frontend.mdHuman writes

Implements full Vanilla JS frontend: CSS, router, API wrapper, auth screens, project list, task list with filters. All backend must be complete first.

Why This Approach Wins

01

Zero Ambiguity

Claude Code never guesses. Every endpoint, field, validation rule, and error code is explicitly written. The output quality equals the spec quality.

02

Reproducible

Run the same task file twice and get the same code. The spec is deterministic. No verbal instructions that are forgotten or misremembered.

03

No Repetition

Generic rules live in CLAUDE.md once. Module rules live in specs once. Task files just reference them. No copy-pasting causes no drift.

04

Easy to Fix

When something is wrong in code, you find the responsible spec, fix one line, re-run the task. The fix propagates automatically.

05

Token Efficient

Claude Code only gets what it needs per task. Auth task never sees tasks spec. Frontend task never sees auth spec. Clean and focused.

06

Team Readable

Any developer can read the specs and understand the entire system without running a single line of code. The spec IS the documentation.

Why No Separate Architecture Document?

⚠️ You already have one — it's called TECHNICAL_SPEC.md

A common misconception is that architecture documents are separate from technical specs. In this project docs/TECHNICAL_SPEC.md is the architecture document. It contains everything a traditional architecture doc would have:

  • 📐 Entity Relationship Diagram — all 5 Mongoose schemas with fields and relations
  • 🗂 Folder Structure — exact file layout for backend and frontend
  • 🔄 Auth Flow — step-by-step JWT + refresh token lifecycle
  • 🏛 Layered Architecture — Routes → Controller → Service → Model pattern
  • ⚠️ Error Handling Strategy — RFC 7807 format, AppError class design
  • 🌍 Environment Config — all env vars and how they are loaded
  • 🗃 Database Indexes — all MongoDB indexes for performance
  • 📋 Module Build Order — dependency sequence for Claude Code

Creating a separate "architecture.md" would duplicate this content and create drift. One file, one truth.
